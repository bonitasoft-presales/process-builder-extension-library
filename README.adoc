A reusable Java 17 library that centralizes enums, constants, records, validators, utilities and the REST execution engine shared across the *Process Builder* ecosystem: actor filters, connectors, REST API extensions and process diagrams. Its goal is to guarantee *code reuse*, *immutability* and *consistency* across every Bonita customization produced by the Presales team.
--

== 1. Problem It Solves

Presales and customer projects routinely duplicate the same building blocks: JSON validation helpers, BDM audit utilities, identity and membership lookups, REST authentication logic, template substitution, email recipient resolution, password cryptography, exception translation. Duplicated code drifts, ages unevenly and becomes a liability during upgrades. This library consolidates those building blocks behind a stable, versioned contract so that every downstream project -- in particular link:https://github.com/bonitasoft-presales/bwh-rest-execution-connector[bwh-rest-execution-connector] -- consumes a single, tested source of truth.

== 2. Features

=== 2.1. Functional Areas

[cols="1,3",options="header"]
|===
| Package | Purpose
| `com.bonitasoft.processbuilder.constants`
| Project-wide constants and schema keys (e.g. `Constants`, `SchemaConstants`).

| `com.bonitasoft.processbuilder.enums`
| Strongly typed enums describing Process Builder concepts: `ActionType`, `BDMObjectType`, `ConfigurationType`, `RestAuthenticationType`, `RestHttpMethod`, `RestOAuth2ClientAuthMethod`, `RestContentType`, `RestApiKeyLocation`, `SmtpType`, `ThemeType`, and more.

| `com.bonitasoft.processbuilder.execution`
| REST execution engine: `ConnectorExecutionEngine`, `ConnectorRequest`, `ConnectorResponse`, `HttpExecutor`, `AuthPipeline`, `TemplateSubstitution`. This is the backbone of the BWH REST Execution Connector.

| `com.bonitasoft.processbuilder.extension`
| Bonita-facing helpers: `BDMAuditUtils`, `ConfigurationUtils`, `EmailRecipientsHelper`, `ExceptionUtils`, `ExecuteSqlDatasource`, `IdentityUtils`, `InputValidationUtils`, `InvolvedUsersParser`, `JsonNodeUtils`, `JsonValidationUtils`, `MembershipUtils`, `PasswordCrypto`.

| `com.bonitasoft.processbuilder.extension.template`
| Configuration templates and authentication catalog used by the execution engine (`template/auth`, `template/catalog`).

| `com.bonitasoft.processbuilder.mapper`
| DTO/POJO mappers between runtime payloads and typed records.

| `com.bonitasoft.processbuilder.records`
| Immutable Java 17 `record` types representing configurations, responses and domain objects.

| `com.bonitasoft.processbuilder.utils`
| General-purpose stateless utilities.

| `com.bonitasoft.processbuilder.validation`
| JSON Schema and domain validators shared by every Process Builder extension.
|===

=== 2.2. Engineering Principles

* *Java 17 first-class*: Records, Sealed Classes, Pattern Matching.
* *Immutability by default* for every value-carrier type.
* *100% test coverage goal* -- every class ships with `*Test` (JUnit 5) and `*PropertyTest` (jqwik) twins.
* *Mutation testing* with PIT (STRONGER mutators in the `full-qa` profile).
* *Static analysis* via Checkstyle, PMD and SpotBugs integrated into the Maven site.

== 4. Consuming the Library

The artifact is published to the *GitHub Package Registry* of the `bonitasoft-presales` organization.

=== 4.1. Maven coordinates

[source,xml,subs="+attributes"]
----
<dependency>
    <groupId>com.bonitasoft.processbuilder.extension</groupId>
    <artifactId>process-builder-extension-library</artifactId>
    <version>{artifact-version}</version>
</dependency>
----

=== 4.2. Authenticating against GitHub Packages

Add a `<server>` entry to `~/.m2/settings.xml` using a GitHub Personal Access Token with the `read:packages` scope:

[source,xml]
----
<servers>
    <server>
        <id>github</id>
        <username>YOUR_GITHUB_USERNAME</username>
        <password>YOUR_GITHUB_TOKEN</password>
    </server>
</servers>
----

Then declare the repository in your POM:

[source,xml]
----
<repositories>
    <repository>
        <id>github</id>
        <url>https://maven.pkg.github.com/bonitasoft-presales/process-builder-extension-library</url>
    </repository>
</repositories>
----

=== 4.3. Using the JAR inside Bonita Studio

To test a local build inside a Bonita project, import the generated `target/process-builder-extension-library-<version>.jar` as a project dependency. Refer to the official guide on link:https://documentation.bonitasoft.com/bonita/latest/managing-extension-studio[managing Studio extensions].

== 5. Quick Start -- Minimal Usage

=== 5.1. Validating a JSON payload

[source,java]
----
import com.bonitasoft.processbuilder.extension.JsonValidationUtils;

String schema  = "{ \"type\": \"object\", \"required\": [\"id\"] }";
String payload = "{ \"id\": 42 }";

boolean valid = JsonValidationUtils.isValid(payload, schema);
----

=== 5.2. Executing a configuration-driven REST call

[source,java]
----
import com.bonitasoft.processbuilder.execution.ConnectorExecutionEngine;
import com.bonitasoft.processbuilder.execution.ConnectorRequest;
import com.bonitasoft.processbuilder.execution.ConnectorResponse;

ConnectorRequest  request  = ConnectorRequest.fromJson(configJson);
ConnectorResponse response = new ConnectorExecutionEngine().execute(request);

if (response.success()) {
    // consume response.body(), response.statusCode(), response.headers()
}
----

NOTE: The fluent API above is the canonical entry point used by `bwh-rest-execution-connector`. Consult the generated Javadoc for the authoritative signatures.

== 6. Project Versioning

The project version lives in the `<version>` tag of `pom.xml` (currently `{artifact-version}`). It follows a `<major>.<minor>.<patch>.<build>` scheme so that rapid internal iterations can be released without inflating the public semver.

Releases are driven by the *Maven Release Plugin* and the `Tag and Release` GitHub Actions workflow:

[source,xml]
----
<plugin>
    <artifactId>maven-release-plugin</artifactId>
    <configuration>
        <tagNameFormat>v@{project.version}</tagNameFormat>
    </configuration>
</plugin>
----

A successful run of the workflow:

. Removes the `-SNAPSHOT` qualifier and commits the release version.
. Tags the commit as `v<version>`.
. Creates a GitHub Release with the packaged JAR attached.
. Publishes the JAR to the GitHub Package Registry.
. Bumps the `pom.xml` to the next `-SNAPSHOT`.

You do *not* need to edit the version manually; trigger `Tag and Release` via `workflow_dispatch` from the Actions tab.

== 9. Troubleshooting

[cols="1,2,2",options="header"]
|===
| Symptom | Likely Cause | Resolution

| `401 Unauthorized` when resolving the dependency
| Missing or wrong GitHub token in `settings.xml`.
| Generate a PAT with `read:packages` and update your Maven `<server>` entry.

| `NoClassDefFoundError: com/fasterxml/jackson/...`
| Jackson dependencies are `provided` -- the consumer must supply them.
| Declare Jackson on the consumer's runtime classpath (Bonita runtime already bundles it).

| Tests fail with `InaccessibleObjectException`
| JDK 17 module access.
| Use the provided Surefire config (already includes `--add-opens java.base/java.lang=ALL-UNNAMED`).

| Coverage below threshold
| New class without a matching `*Test` / `*PropertyTest`.
| Add the mandatory twin test files per the AGENTS.md policy.

| Maven site fails with `openapi-generator` errors
| Schema file missing or invalid.
| Verify `src/main/resources/schemas/openapi.yaml`; the plugin runs with `skipValidateSpec=true` but still requires the file to exist.
|===

== 10. Contributing

We follow *Clean Code* and *Git Flow*. Branch naming and PR flow:

. Create a branch from `master` using the Jira ticket ID:
+
[source,bash]
----
git checkout -b <jira-ticket-id>_<short-description> master
----
. If `develop` does not exist yet, create and push it:
+
[source,bash]
----
git checkout -b develop master
git push origin develop
----
. Run the full verification locally:
+
[source,bash]
----
./mvnw clean install site
----
. Open a Pull Request against `develop`. Make sure:

* Every new or modified class has `*Test.java` *and* `*PropertyTest.java` twins.
* Line coverage >= 95% (target 100%), branch coverage >= 95%, mutation coverage >= 85%.
* No regressions reported by PMD, Checkstyle or SpotBugs.
